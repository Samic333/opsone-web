# Safety Report Template Model

## Design Philosophy

### Why Not Hard-Coded Silos

A naive implementation of a multi-type safety reporting system creates a separate table, controller, and form per report type. In a 9-type system this means:

- 9 tables with largely duplicate columns
- 9 sets of controller actions sharing identical validation and submission logic
- 9 iPad form views that must all be updated whenever a base field changes
- Schema migrations for every new type

The OpsOne template model avoids this by maintaining a **single `safety_reports` table** for all types. Every report — regardless of type — shares a common set of base fields mapped to real database columns. Type-specific data (fields that only apply to, say, a Flight Crew Occurrence or a Maintenance Engineering report) is stored as JSON in the `extra_fields` column.

This approach means:

- One controller, one model, one submission endpoint
- Base field validation is written once
- Adding a new report type requires no schema migration in most cases
- The iPad and web portal read the same data structure

### `extra_fields` JSON Column

`safety_reports.extra_fields` is a `JSON` column (MySQL 8) / `TEXT` column (SQLite, parsed by application layer). It stores a flat key-value map of type-specific field values. The application layer is responsible for knowing which keys are expected for a given `report_type`.

The `SafetyReportModel` does not validate `extra_fields` schema at the database level — validation occurs in the controller before insert.

### Template Versioning

`safety_reports.template_version` is a `TINYINT UNSIGNED NOT NULL DEFAULT 1`. If the expected structure of `extra_fields` for a given type changes (e.g., a field is renamed or a new required field is added), the version is incremented. Existing reports retain their original `template_version` value and are rendered using the matching legacy schema. This allows the form UI to evolve without breaking historical records.

## Base Fields

These columns are present on the `safety_reports` table for all report types. They are captured on every submission regardless of type.

| Field | Column | Type | Required |
|---|---|---|---|
| Report Title | `title` | `VARCHAR(255)` | Yes |
| Event Date | `event_date` | `DATE` | Yes |
| UTC Time | `event_utc_time` | `TIME` | No |
| Local Time | `event_local_time` | `TIME` | No |
| Location | `location_name` | `VARCHAR(255)` | No |
| ICAO Code | `icao_code` | `VARCHAR(10)` | No |
| Occurrence Type | `occurrence_type` | `ENUM('accident','serious_incident','incident','hazard','observation')` | Yes |
| Event Type | `event_type` | `VARCHAR(100)` | No |
| Initial Risk Score | `initial_risk_score` | `TINYINT UNSIGNED` | No |
| Narrative | `description` | `TEXT` | Yes |
| Reporter Position | `reporter_position` | `VARCHAR(100)` | No |
| Anonymous | `is_anonymous` | `TINYINT(1)` | No (default 0) |
| Report Type | `report_type` | `VARCHAR(50)` | Yes (system-set) |
| Reference Number | `reference_no` | `VARCHAR(50)` | Yes (system-generated) |
| Status | `status` | `VARCHAR(50)` | Yes (default `draft`) |
| Severity | `severity` | `VARCHAR(20)` | No (default `unassigned`) |
| Template Version | `template_version` | `TINYINT UNSIGNED` | Yes (default `1`) |
| Extra Fields | `extra_fields` | `JSON` | No (type-specific) |
| Reporter | `reporter_id` | `INT UNSIGNED FK → users` | Yes (system-set, NULL if anonymous) |
| Assigned To | `assigned_to` | `INT UNSIGNED FK → users` | No |
| Tenant | `tenant_id` | `INT UNSIGNED FK → tenants` | Yes (system-set) |

**System-set fields** are populated by the controller or model and are never accepted from user input directly.

**Reference number format:** `{TYPE_UPPER}-{YEAR}-{NNNN}` — e.g., `FLIGHT_CREW_OCCURRENCE-2026-0003`. Generated by `SafetyReportModel::generateReference()` using a `COUNT`-based sequence per tenant per year.

## Report-Specific Field Groups

### Aircraft Section (used by: Flight Crew Occurrence, TCAS, Maintenance Engineering, FRAT)

Stored in `extra_fields`. Captures the aircraft involved in the occurrence.

| Field | Key | Notes |
|---|---|---|
| Aircraft Registration | `aircraft_registration` | e.g., `A6-OPS` |
| Aircraft Type | `aircraft_type` | e.g., `Boeing 737-800` |
| Flight Number | `flight_number` | e.g., `OPS101` |
| Departure Airport | `departure_airport` | ICAO or IATA code |
| Destination Airport | `destination_airport` | ICAO or IATA code |
| Phase of Flight | `phase_of_flight` | e.g., `climb`, `cruise`, `approach`, `landing` |

### Flight Crew Section (used by: Flight Crew Occurrence, TCAS)

| Field | Key | Notes |
|---|---|---|
| Captain Name | `captain_name` | Free text |
| First Officer Name | `fo_name` | Free text |
| Captain Role | `captain_role` | e.g., `PIC`, `PM` |
| FO Role | `fo_role` | e.g., `PF`, `PM` |
| Cabin Manager | `cabin_manager` | Free text |
| Flight Attendant Count | `flight_attendant_count` | Numeric string |
| Aircraft Engineer | `aircraft_engineer` | Free text |

### Maintenance Section (used by: Maintenance Engineering)

| Field | Key | Notes |
|---|---|---|
| Engineering Finding | `engineering_finding` | Free text narrative |
| Recommendations | `recommendations` | Free text |
| Part Number (on) | `part_number_on` | Installed part number |
| Part Number (off) | `part_number_off` | Removed part number |
| Serial Number (on) | `serial_number_on` | |
| Serial Number (off) | `serial_number_off` | |
| Batch Number | `batch_number` | |
| Defect Sheet Number | `defect_sheet_number` | AMO/MRO defect reference |

### FRAT Section (used by: FRAT)

The FRAT (Flight Risk Assessment Tool) is a scored checklist. In addition to the Aircraft Section, it captures:

| Field | Key | Notes |
|---|---|---|
| Departure Station | `departure_station` | ICAO/IATA |
| Destination Station | `destination_station` | ICAO/IATA |
| Planned Departure Time | `planned_departure_time` | ISO 8601 UTC |
| Crew Rest Assessment | `crew_rest` | `adequate`, `marginal`, `inadequate` |
| Crew Fatigue Level | `crew_fatigue_level` | `none`, `low`, `moderate`, `high` |
| Weather Concern | `weather_concern` | `none`, `moderate`, `significant`, `severe` |
| Additional Risk Factors | `additional_risk_factors` | Free text |
| Computed Risk Score | `frat_score` | Integer — computed by iPad `FRATFormView` |
| Risk Level | `frat_risk_level` | `low`, `caution`, `high` |
| Base Manager Approval Required | `base_manager_approval_required` | `true`/`false` — set when score >= 25 |

The iPad `FRATFormView` computes `frat_score` by summing the `score` values of all selected `FRATItem` entries before submission. The risk thresholds are: `< 15` = Low, `15–24` = Caution, `>= 25` = High (Base Manager approval required).

**Ground Ops** emphasises `location_name` and `icao_code` from the base fields; no additional `extra_fields` sections.

## Adding a New Report Type

Follow these steps when introducing a new report type to the system.

1. **Add slug and label to `SafetyReportModel::TYPES`:**
   ```php
   'new_type_slug' => 'New Type Display Label',
   ```

2. **Define role visibility in `SafetyReportModel::TYPE_ROLES`:**
   ```php
   'new_type_slug' => ['pilot', 'cabin_crew', 'safety_officer', 'airline_admin'],
   ```
   Only roles in this list will see the type in the submission selector or on the iPad tile grid.

3. **Add type-specific form section in `app/Views/safety/report_form.php`:**
   Add a PHP conditional block for `$reportType === 'new_type_slug'` that renders the additional fields.

4. **Add a dedicated SwiftUI form view or extend `GeneralSafetyFormView` in `CrewAssist/Features/Reports/`:**
   If the type needs a specialist checklist (like FRAT) or structured sections, create a new `NewTypeFormView.swift`. Otherwise, `GeneralSafetyFormView` handles submission and only the `SafetyReportsListView` tile label needs updating.

5. **Document the `extra_fields` JSON structure in this file** under the relevant section above.

6. **Update `safety_module_settings` default `enabled_types` seed** in `database/seeders/phase1_seed.php` if the type should be enabled by default for all new tenants.

7. **Update `docs/safety-role-access.md`** — add the new type to the Report Type Access Matrix table.

8. **Register retention support** if the new type introduces its own table (unlikely under the template model, but possible for highly structured types): add entries to `RetentionService::DEFAULTS` and `RetentionService::MODULE_TABLE_MAP` per the instructions in `docs/retention-policy-framework.md` Section 9.
